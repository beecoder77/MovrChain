# MovrChain

**The running app that proves you ran — and pays you for it.**

MovrChain is a Monad-native running tracker built for the post-run check-in. It allows runners to import GPX route data, replay their routes with statistics, attest the activity on-chain with zero-overhead finality, publish it to a global community feed, and claim milestone rewards. The app also features an achievement NFT system, staking pools with NFT-backed yield boosts, and community running clubs with decentralized treasuries.

For judges or quick manual verification, the app is pre-configured with active contracts on the Monad Testnet and includes a local sample run simulation.

1. **Install Dependencies**:
   ```bash
   npm install
   ```
2. **Run Development Server**:

   - To test the on-chain attestation and claims, connect a MetaMask/evm wallet configured to the **Monad Testnet**.

## Core Application Flows

MovrChain's user experience is split into distinct, highly interactive phases:

```
[Import GPX] ──► [Animated Route Replay] ──► [Run Summary] ──► [On-Chain Verify & Claim]
```

### 1. Import
Users upload standard GPX files (generated by Strava, Apple Watch, Garmin, etc.) containing GPS coordinates and time telemetry. The file is parsed in the browser via [gpx.ts](file:///Users/macbookpro/Playground/MonadHackathon/src/lib/gpx.ts) to calculate distance, duration, pace, and route points.

### 2. Animated Replay
Once loaded, a stat-focused view plays an animated trace of the run's GPX coordinates on an interactive map. In this screen, distance dominates the screen space, with pace and duration supporting it.

### 3. Summary
The user reviews a static summary of their effort. This screen lists total distance (km), average pace, duration, and the full route layout.

### 4. On-chain Verification & Claims
Upon clicking **"Verify & publish"**, the application walks the user through a sequential blockchain transaction pipeline:
- **Attestation**: The runner submits the run hash and telemetry to the `MovrChainAttestation` contract. The contract checks that the GPX hash is unique, updates the runner's aggregated statistics (total distance, run count, best run, daily run streak), and emits `RunAttested`.
- **Feed Publish**: The run is published to the on-chain community timeline on the `MovrFeed` contract.
- **Milestone Rewards**: If the run is $\ge 1\text{ km}$, the runner can claim $1\text{ MOVR}$ per kilometer from the `MovrMilestoneReward` pool.
- **Additive Club Share**: If the runner is a member of an active on-chain club, an additional reward of $1\text{ MOVR}$ per $10\text{ km}$ is routed directly to the club's smart-contract treasury to back future club-wide challenge payouts.

### 5. Staking & Achievements
- **Achievements**: The `AchievementNFT` contract defines milestone templates (e.g., First 5K, First Half Marathon, 7-Day Streak). When a runner's attested statistics meet these requirements, they can claim a unique NFT badge.
- **Staking**: Users can stake their earned `MOVR` in the `MovrStaking` contract. Holding claimed Achievement NFTs grants a staking yield boost (capped by `maxBoostBps`). Stakers can also choose to donate a percentage of their staking rewards to their club's treasury.

### 6. Running Clubs
Discover and join clubs on the **Clubs** tab. Clubs are ranked by active treasury size and total clubmate runs.
- **Manager Approval**: Private clubs require approval from the club manager to join.
- **Challenges**: Club managers can create runs/streaks challenges with reward pools from the treasury. Members submit proofs to settle rewards.

## Local Development & Frontend Setup

### Setup Environment
The application contains hardcoded, pre-deployed contract addresses in [contracts.ts](file:///Users/macbookpro/Playground/MonadHackathon/src/lib/contracts.ts) pointing to active Monad Testnet contracts. To deploy and run with your own contracts, set up overrides in `.env.local` at the root of the project:

```bash
# Override contract addresses
VITE_CONTRACT_ADDRESS=0xAttestationAddress
VITE_PROFILE_ADDRESS=0xProfileAddress
VITE_MOVR_TOKEN=0xTokenAddress
VITE_ACHIEVEMENT_NFT=0xAchievementNftAddress
VITE_STAKING=0xStakingAddress
VITE_CLUB_REGISTRY=0xClubRegistryAddress
VITE_CLUB_MEMBER_NFT=0xClubMemberNftAddress
VITE_CLUB_BADGE_NFT=0xClubBadgeNftAddress
VITE_CLUB_CHALLENGES=0xClubChallengesAddress
VITE_MILESTONE_REWARD=0xMilestoneRewardAddress
VITE_FEED_ADDRESS=0xFeedAddress
```

### Static Assets Generation
The frontend includes utilities to generate customized athletic avatars and achievement badge SVGs:
- **Generate Avatars**: Generates SVGs under `public/brand/avatars/`
  ```bash
  npm run generate:avatars
  ```
- **Generate Achievement Metadata/Art**: Generates on-chain data URIs and metadata JSON
  ```bash
  npm run generate:art
  ```

## Contract Deployment Guide (Monad Testnet)

Before running the deployment scripts, make sure you configure your local environment.

### 1. Prerequisites & Environment Setup
- Install Monad-compatible Foundry:
  ```bash
  foundryup --network monad
  ```
- Change directory and copy environment template:
  ```bash
  cd contracts
  cp .env.example .env
  ```
- Open `contracts/.env` and configure:
  - `PRIVATE_KEY`: Private key of a funded Monad Testnet deployer wallet.
  - `ADMIN_ADDRESS`: Optional secondary administrator address.

To request gas money for deployment, visit the [Monad Faucet](https://faucet.monad.xyz/).

---

### 2. Deploy Script Pipeline
The deployment scripts run using `--gas-estimate-multiplier 250` and `--slow` options to account for Monad gas estimations and to avoid race conditions.

#### A. Core Stack (Token, Attestation, Achievements, Staking)
Runs deployment, registers default achievements, mints initial reward tokens, and funds the staking rewards pool:
```bash
./deploy.sh
```
*Output prints contract addresses; populate `MOVR_TOKEN`, `ATTESTATION`, `ACHIEVEMENT_NFT`, and `STAKING` in `contracts/.env`.*

#### B. Profile Registry
```bash
./deploy-profile.sh
```

#### C. Clubs & Staking Link
Deploys `ClubMemberNFT`, `MovrClubRegistry`, `ClubBadgeNFT`, and redeploys the staking contract to link club yield donations:
```bash
./deploy-clubs.sh
```
*Output prints registry and member/badge NFT addresses; populate `CLUB_REGISTRY` in `contracts/.env`.*

#### D. Club Challenges
```bash
./deploy-club-challenges.sh
```

#### E. Milestone Rewards & Pool Funding
Deploys reward router, approves the router to draw tokens, and funds the reward pool with `100,000 MOVR`:
```bash
./deploy-milestone-reward.sh
```

#### F. Community Feed
```bash
./deploy-feed.sh
```

## Administrative & Operations Scripts

A set of shell scripts is available under `contracts/` to manage, update, and audit active deployments.

### Staking Pool Funding
To fund the `MovrStaking` reward pool with additional `MOVR` tokens (automatically mints more if owner balance is insufficient):
```bash
./fund-rewards.sh
```
*Adjust reward quantity by overriding env variables: `REWARD_AMOUNT=500000000000000000000000 ./fund-rewards.sh`.*

### Checking Roles
Audit owner (`DEFAULT_ADMIN_ROLE`) and admin (`ADMIN_ROLE`) configurations on active contracts:
```bash
./check-role.sh                 # Inspects address defined in PRIVATE_KEY
./check-role.sh 0xYourAddress   # Inspects a specific address
```

### Minting MOVR
Directly mint `MOVR` tokens to a target address:
```bash
./mint-movr.sh 10000000         # Mints 10,000,000 MOVR to the deployer wallet
```

### Achievements Art & URI Updates
If achievement SVGs or metadata text have changed, you can update them without redeploying the core contracts:
1. Re-generate SVGs and metadata files from the root of the project: `npm run generate:art`.
2. Push updates to the contracts:
   ```bash
   # Redeploys AchievementNFT and Staking contracts with fresh data-URIs
   ./redeploy-nft-staking.sh

   # Incremental URI updates only (calls setAchievementURI)
   ./update-achievement-uris.sh
   ```

## Monad Testnet Network Details

The application is optimized for the following network details:

* **Chain ID**: `10143`
* **RPC URL**: `https://testnet-rpc.monad.xyz`
* **Currency Symbol**: `MON`
* **Block Explorer**: `https://testnet.monadvision.xyz` or Sourcify details.

### Gas Limit Considerations
Monad charges on the transaction's **gas limit** rather than just the gas consumed. Standard gas estimation APIs like `eth_estimateGas` can underestimate gas usage for contract calls that write heavy data (such as appending strings to arrays in `MovrFeed` or updating large struct maps). 

To ensure transactions do not fail due to Out-Of-Gas (OOG) errors:
- Deployment scripts use a 2.5x multiplier (`--gas-estimate-multiplier 250`).
- Frontend calls apply a 2x buffer (`GAS_BUFFER_BPS = 200n`) with safety floors:
  - Attestation: `350,000` gas
  - Feed Publish: `800,000` gas
  - Milestone Claim: `550,000` gas
